## 2. 三端口不混乱的硬规则（必须遵守）
### 2.1 路由与动态参数统一
- 同一个动态路径只能有一个 slug 名称：例如只能用 `[id]`，禁止同时存在 `[eventId]` 与 `[id]` 指向同路径（否则 Next/Vercel 会报 slug 冲突）。

### 2.2 共享代码边界
- `apps/admin` 只能引用：
  - `packages/shared/*`
  - `apps/admin/*`
- `apps/internal-web` 只能引用：
  - `packages/shared/*`
  - `apps/internal-web/*`
- `apps/customer-web` 只能引用：
  - `packages/shared/*`
  - `apps/customer-web/*`
- 禁止跨端口 import（例如 customer 引用 admin 的组件）。

### 2.3 数据源统一
- 活动只从 `events_v2` 读取与写入。
- 票种只从 `ticket_types_v2`（如已定）读取与写入。
- 周配置只从 `event_weeks` + `event_week_days` 读取与写入。
- 若发现旧表仍被引用，必须替换为 V2 查询并删除旧代码路径。

## 3. Events V2 统一规范（必须做到）
### 3.1 事件状态与发布逻辑
- Event 的状态字段（如 `status`）必须是统一枚举（active/temp_closed/archived/draft），三端口一致。
- “发布”必须有明确动作：
  - draft -> active（发布）
  - active -> archived（下架）
  - active -> temp_closed（临时关闭）
- 管理端必须提供 publish/unpublish UI，并调用统一 API。

### 3.2 事件可见性规则（customer）
- customer 端查询必须只返回当前 region 下可见事件：
  - 以 `region_id` 作为第一筛选（从用户选择的 region）
  - 再按 event.status=active
- 如果 event 没有 region/venue 绑定，必须在创建时自动补齐（见 3.3）。

### 3.3 创建事件时的自动绑定（必须实现）
- 创建 `events_v2` 时，必须保证：
  - event.merchant_id 有值
  - event.venue_id / region_id（选一种作为主筛选）必须可推导并写入
- 推荐：event 直接存 `venue_id` + 冗余 `region_id`（避免复杂 join）。
- 创建成功后必须跳转到成功页或配置页（不允许“创建后无跳转”）。

## 4. 数据库修改与自动 db push（强制）
### 4.1 迁移原则
- **所有 schema 变更必须通过迁移**（SQL migration / Prisma migration / Supabase migration，按项目现状选一种）。
- 改完 schema 后必须触发 db push，确保本地与远端一致。

### 4.2 自动化要求
- 若项目使用 Prisma：
  - 修改 schema.prisma 后必须执行：
    - `prisma migrate dev`（开发）或 `prisma migrate deploy`（生产）
    - `prisma db push`（若你们规范需要）
- 若项目使用 Supabase migrations：
  - 在 `supabase/migrations` 生成迁移文件
  - 执行 `supabase db push` 或你们的 CI 命令
- **每次 DB 变更后，你必须在回复中输出：**
  - 迁移文件名
  - 执行的命令
  - 变更的表/字段列表

## 6. PostgREST embed 冲突（PGRST201）处理规范（必须遵守）
当出现：
`Could not embed because more than one relationship was found...`
说明 merchants ↔ venues 存在多条关系路径。

### 处理策略（必须选其一并落实）
A) **显式指定外键关系**（推荐）
- 查询中使用 `venues!<fk_name>(...)` 或 `merchants!<fk_name>(...)`
- 避免 PostgREST 自动猜关系导致报错

B) **重构数据模型**
- 保留唯一关系路径（例如 events_v2 直接存 venue_id）
- 移除多余的 FK 或改名避免歧义

### 输出要求
- 每次修复该类问题，你必须在回复中写清楚：
  - 触发冲突的原查询
  - 最终采用的 FK 指定写法或模型调整
  - 为什么这样不会再冲突
